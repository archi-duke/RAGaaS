# 구조적 의사결정 목록

## 개요

### 목적
명세된 구조 설계서(architecture.md)에서 구조적 의사결정을 식별하고 분석하여, 설계된 구조가 어떤 설계 결정에 기반하고 있는지, 각 결정이 어떤 품질 요구사항에 영향을 미치는지를 명확히 문서화합니다.

### 분석 범위
- **분석 대상**: architecture.md의 2장(요구사항), 3장(시스템 구조), 4장(모듈 구조)
- **제외**: 부록은 참고하지 않음
- **식별 기준**: 
  - 대안을 생각할 수 있는 설계 결정
  - 중요한 품질 요구사항(QA)에 영향을 미치는 설계 결정

## 요구사항 요약

### 기능적 요구사항
- **ASR-101: 하이브리드 검색 수행**: 벡터(ANN)와 키워드(BM25) 결과를 결합하여 정밀한 컨텍스트 제공.
- **ASR-103: 저장소 오케스트레이션**: 이종 DB(Vector, Graph, Meta) 자원 관리.
- **ASR-203: 대용량 데이터 확장성**: 대규모 청크 처리 및 인덱싱.
- **ASR-204: 검색 파이프라인 실험**: 동적 파라미터 기반 실험 환경 제공.

### 품질 요구사항

#### 품질 속성 (QA)
- **QA-001-성능**: 검색 지연 시간 최소화 및 고속 인덱싱. (우선순위: 1)
- **QA-002-변경 용이성**: 엔진 및 모델 교체 비용 최소화. (우선순위: 2)
- **QA-003-확장성**: 데이터 및 부하 증가에 따른 수평적 확장. (우선순위: 3)
- **QA-004-사용성**: 실험 설정 시 즉각적인 피드백 및 유연성. (우선순위: 4)

## 구조적 의사결정 목록

### AD-001: 헥사고날 아키텍처 (Ports & Adapters) 적용

- **의사결정 내용**: 도메인 로직을 외부 기술(DB, API)로부터 격리하기 위해 Port-Adapter 계층 구조를 채택함.
- **관련 구조**: 4장 4.1 모듈 명세 및 4.3 폴더 구조
- **대안**: 
  - **대안 1**: 전통적인 계층형 아키텍처 (Layered Architecture). 인프라에 직접 의존.
- **영향받는 품질 요구사항**:
  - **QA-002 (변경 용이성)**: (++) - 특정 DB나 모델 엔진 교체 시 도메 로직 수정 없이 어댑터만 교체 가능.
  - **QA-001 (성능)**: (0) - 약간의 추상화 오버헤드가 있으나 무시할 수 있는 수준임.
- **트레이드오프**: 초기 개발 복잡도가 증가하지만, 장기적인 유지보수성과 기술 스택 교체 유연성을 확보함.

### AD-002: 검색 엔진 전략 패턴 (Strategy Pattern) 채택

- **의사결정 내용**: 다양한 검색 알고리즘(ANN, Keyword, Hybrid)을 동적으로 전환할 수 있도록 전략 패턴으로 캡슐화함.
- **관련 구조**: 3장 3.1 컴포넌트 명세 및 4장 4.1 모듈 명세
- **대안**: 
  - **대안 1**: 하드코딩된 조건 분기 (If-Else).
  - **대안 2**: 정적 파이프라인 흐름 제어 (CA-102).
- **영향받는 품질 요구사항**:
  - **QA-004 (사용성)**: (++) - 플레이그라운드에서 실시간으로 검색 로직을 전환/변경하는 데 필수적임.
  - **QA-002 (변경 용이성)**: (+) - 새로운 검색 알고리즘 추가 시 기존 코드 변경 없이 전략 클래스만 추가 가능.
- **트레이드오프**: 코드의 분산으로 인해 흐름을 한눈에 파악하기 다소 어려울 수 있으나 실험 유연성을 극대화함.

### AD-003: 비동기 병렬 검색 (Parallel Retrieval) 수행

- **의사결정 내용**: 하이브리드 검색 시 벡터 검색과 키워드 검색을 비동기로 동시에 실행하여 응답 시간을 단축함.
- **관련 구조**: 3장 3.3 품질 설계 (CA-201)
- **대안**: 
  - **대안 1**: 순차적(Sequential) 검색 수행.
- **영향받는 품질 요구사항**:
  - **QA-001 (성능)**: (++) - Critical Path가 가장 긴 검색 작업 시간으로 수렴되어 전체 레이턴시가 대폭 감소함.
- **트레이드오프**: 동시 리소스 점유율이 높아지므로 Throttling(CA-201A)과 같은 제어 기작이 동반되어야 함.

### AD-004: 메시지 큐 기반 대용량 인제스션 분산

- **의사결정 내용**: 무거운 문서 처리(파싱, 임베딩) 작업을 메시지 큐를 통해 워커 클러스터로 이관함.
- **관련 구조**: 3장 3.2 동작 명세 (CA-203)
- **대안**: 
  - **대안 1**: API 서버 내에서 동기적 또는 쓰레드 기반 비동기 처리.
- **영향받는 품질 요구사항**:
  - **QA-003 (확장성)**: (++) - 워커 수를 조절하여 수평적 확장이 용이함.
  - **QA-001 (성능)**: (+) - API 서버가 즉각 응답(202 Accepted)을 반환하여 사용자 체감 성능 향상.
- **트레이드오프**: 메시지 브로커 운영 비용과 시스템 복잡도가 증가함.

### AD-005: 네임스페이스 기반 물리적 인덱스 파티셔닝

- **의사결정 내용**: 지식 베이스(KB)별로 독립된 컬렉션/네임스페이스를 생성하여 데이터를 물리적으로 격리함.
- **관련 구조**: 3장 3.1 컴포넌트 명세 (CA-106)
- **대안**: 
  - **대안 1**: 단일 거대 인덱스에 KB_ID 필드를 통한 논리적 필터링 검색.
- **영향받는 품질 요구사항**:
  - **QA-003 (확장성)**: (++) - 특정 KB의 데이터 증가가 다른 KB 검색 성능에 영향을 주지 않음.
  - **QA-001 (성능)**: (+) - 검색 대상 데이터 셋이 제한되어 ANN 검색 속도가 향상됨.
- **트레이드오프**: 관리해야 할 컬렉션 수가 많아져 메타데이터 관리 부하가 발생할 수 있음.

### AD-006: 후처리 인터셉터 체인 (Interceptor Chain) 적용

- **의사결정 내용**: 리랭킹, 필터링 등 검색 후처리 단계를 Chain of Responsibility 패턴으로 구성함.
- **관련 구조**: 4장 4.1 모듈 명세 (CA-104)
- **대안**: 
  - **대안 1**: 단일 서비스 클래스 내의 거대 함수.
- **영향받는 품질 요구사항**:
  - **QA-002 (변경 용이성)**: (++) - 새로운 필터 추가나 기존 필터 순서 변경이 매우 자유로움.
  - **QA-004 (사용성)**: (+) - 파라미터에 따라 특정 필터를 즉시 On/Off 하기 용이함.
- **트레이드오프**: 필터가 많아질 경우 각 필터 간의 데이터 의존성 관리가 복잡해질 수 있음.
