# RAG 시스템의 딜레마: 재현율(Recall) vs 정밀도(Precision), 그리고 Fallback의 역할

RAG(Retrieval-Augmented Generation) 시스템을 구축하다 보면 엔지니어는 항상 딜레마에 빠집니다. **"답을 못 찾는 것보다 뭐라도 가져오는 게 나을까, 아니면 틀린 정보를 가져오느니 모른다고 하는 게 나을까?"**

최근 RAGaaS 프로젝트에서 그래프 검색(Graph Search)의 **Fallback(대체 검색) 정책**을 두고 깊은 고민을 했고, 그 결정 과정을 공유합니다.

## 1. 문제 상황: "스승의 스승을 찾아라"

사용자가 "성기훈의 스승의 스승은 누구야?"라는 질문을 던졌습니다. 이 질문을 해결하려면 그래프의 **방향성**이 생명입니다.
- `(성기훈)-[:스승]->(A)-[:스승]->(B)`

하지만 데이터에는 반대 방향인 `제자` 관계만 있을 수도 있습니다. 이를 해결하기 위해 **Inverse Search(역방향 추론)** 기능을 도입했습니다.
- `(B)-[:제자]->(A)-[:제자]->(성기훈)`

문제는 사용자가 **"역방향 추론을 하지 마라(Strict Mode)"**라고 옵션을 껐을 때 발생했습니다.

## 2. Fallback의 함정

LLM이 엄격한 Cypher/SPARQL 쿼리 생성에 실패했을 때, 기존 시스템은 **Fallback 로직**을 가동했습니다.
- **로직**: 질문에 포함된 키워드("성기훈", "스승")가 포함된 트리플을 정규식(Regex)으로 긁어옴.
- **결과**: 방향성이 무시된 트리플들이 검색 결과에 포함됨.
  - 예: `(성기훈)-[:제자]->(누군가)` (질문은 스승을 원했는데 제자가 검색됨)

이로 인해 사용자는 "역방향 검색을 껐는데 왜 결과가 나오지? (심지어 틀릴 수도 있는 결과가)"라는 혼란을 겪게 됩니다.

## 3. 하이브리드 RAG에서의 역할 분담

우리는 여기서 **Hybrid RAG**의 구조적 이점을 다시 생각했습니다.

1.  **Vector/BM25 검색**: 문맥적 유사성을 기반으로 **넓은 범위의 정보(High Recall)**를 담당합니다. 이미 여기서 "관련 있어 보이는" 텍스트는 충분히 수집됩니다.
2.  **Graph Search**: 구조적 연결을 통해 **정밀한 논리적 추론(High Precision)**을 담당해야 합니다.

그래프 검색이 쿼리 생성에 실패했다고 하여 단순 키워드 매칭(Fallback)으로 전환하는 것은, 그래프 검색 스스로의 강점(정밀함)을 버리고 Vector 검색의 하위 호환으로 전락하는 것과 같습니다. 이는 검색 결과에 **노이즈(Noise)**만 추가할 뿐입니다.

## 4. 우리의 결정: "모르는 건 모른다고 하자"

우리는 다음과 같은 정책을 확립했습니다.

> **"사용자가 엄격한 검색(Inverse Off)을 요청했다면, LLM 쿼리 실패 시 Fallback을 수행하지 않고 빈 결과를 반환한다."**

### 변경된 동작 방식
1.  **Inverse Mode ON**: LLM이 적극적으로 역방향 패턴(`|`, `^`)을 사용하여 추론을 시도합니다.
2.  **Inverse Mode OFF**:
    *   LLM에게 **"반드시 정방향 화살표(`->`)만 사용하라"**고 강제합니다.
    *   쿼리 결과가 없으면 Fallback을 타지 않고 종료합니다.

### 기대 효과
*   **환각(Hallucination) 방지**: 맥락 없는 트리플이 LLM의 추론을 방해하는 것을 막습니다.
*   **사용자 의도 존중**: 사용자가 제약 조건을 걸었을 때, 시스템이 이를 무시하고 "과도한 친절"을 베푸는 것을 방지합니다.
*   **시스템 신뢰도 향상**: 될 때도 있고 안 될 때도 있는 "운"에 맡기는 검색이 아니라, 논리에 기반한 예측 가능한 검색 시스템이 됩니다.

## 5. 결론

RAG 시스템 고도화는 단순히 "더 많은 정보"를 가져오는 것이 아니라, **"각 검색 모듈이 제 역할을 정확히 수행하는 것"**에 있습니다. 때로는 과감하게 정보를 버리는 것이 정확한 답변을 위한 지름길입니다.
