# RAGaaS 코드 구조 분석

이 문서는 RAGaaS 프로젝트의 전체 코드 구조를 분석·정리한 결과입니다.  
새로 합류한 개발자나 기능 확장 시 참고용으로 사용하세요.

---

## 1. 프로젝트 개요

- **목적**: Milvus 기반 다수의 RAG 지식 베이스 생성·관리
- **역할**:
  - 문서 업로드 → 청킹 → 임베딩 → Vector DB(Milvus) + Graph DB(Fuseki/Neo4j) 적재
  - 다양한 검색 전략(ANN, BM25, Hybrid, Graph RAG)으로 청크 검색
- **기술 스택**: FastAPI, React(Vite/TS), MongoDB(Beanie), Milvus, Fuseki/Neo4j, OpenAI Embedding

---

## 2. 전체 디렉터리 구조

```
RAGaaS/
├── backend/                    # FastAPI 메인 백엔드 (실제 진입: backend/main.py)
│   ├── main.py                 # 앱 진입점, 라우터 등록, startup(Mongo/Milvus/프롬프트 시드)
│   ├── app/
│   │   ├── api/                # HTTP 라우터
│   │   ├── core/               # 설정, DB/벡터/그래프 클라이언트
│   │   ├── models/             # Beanie 도큐먼트 모델
│   │   ├── schemas/            # Pydantic 요청/응답 스키마
│   │   └── services/           # 비즈니스 로직 (인제스션, 검색)
│   └── data/prompts/           # 그래프 추출/쿼리용 프롬프트
├── frontend/                   # React + Vite + TypeScript
│   └── src/
│       ├── pages/              # Dashboard, KnowledgeBaseDetail, GraphViewer
│       ├── components/        # 모달, 탭, 검색 UI 등
│       ├── services/api.ts     # Backend API 래퍼 (kbApi, docApi, retrievalApi 등)
│       └── hooks/              # useWebSocket 등
├── ingest_service/             # 별도 FastAPI 서비스 (LlamaIndex 기반 문서 인제스션)
│   └── app/
│       ├── api/ingest.py       # /api/ingest 등
│       ├── core/               # pipeline, fuseki/neo4j/milvus 연결, 텍스트 정제 등
│       └── main.py
├── data/uploads/               # 공유 스토리지 (백엔드·ingest_service가 공유)
└── docs/                       # 사양, 아키텍처, 배포 가이드
```

**실제 사용되는 진입점**: `backend/main.py` (루트 `main.py`).  
`backend/app/main.py`는 별도 앱 스켈레톤으로 보이며, **uvicorn main:app** 시에는 `backend/` 디렉터리 기준 `main.py`가 로드됩니다.

## 3. 백엔드(backend) 상세

### 3.1 진입점과 라우팅 (`backend/main.py`)

- **FastAPI 앱** 생성, CORS, Validation 에러 핸들러
- **라우터 등록**:
  - `knowledge_base.router` → `/api/knowledge-bases`
  - `document.router` → `/api/knowledge-bases` (문서 CRUD, 업로드)
  - `retrieval.router` → `/api/knowledge-bases` (검색)
  - `graph_viewer.router` → `/api/retrieval/graph`
- **WebSocket**: `/api/ws/{kb_id}` — 문서 처리 상태 실시간 푸시
- **Startup**:
  - MongoDB(Beanie) 초기화
  - Milvus 연결
  - `graph_extraction_prompt` 시드(없으면 파일에서 로드 후 DB 저장)

### 3.2 API 레이어 (`app/api/`)

| 파일 | 역할 |
|------|------|
| `knowledge_base.py` | KB CRUD, promote, extraction-rules, extraction-prompt, query-prompt, schema 추출 등 |
| `document.py` | 문서 업로드, 삭제, 청크 조회, 파이프라인 상태/데이터, 청크 수정 |
| `retrieval.py` | `POST /{kb_id}/retrieve` — 검색 실행 (전략/파이프라인/그래프/reranker 등) |
| `graph_viewer.py` | 그래프용 triples/chunk 등 조회 |
| `websocket_endpoint.py` | (app/main.py 쪽에서 사용될 수 있는 WS 라우터) |

- 문서 업로드(`document.py`)는 파일을 `SHARED_STORAGE_PATH`에 저장한 뒤 **Ingest Service**를 비동기로 호출합니다.

### 3.3 코어 (`app/core/`)

| 파일 | 역할 |
|------|------|
| `config.py` | Pydantic Settings — MONGO_URI, MILVUS_*, FUSEKI_URL, NEO4J_*, OPENAI_API_KEY, INGEST_SERVICE_URL, SHARED_STORAGE_PATH |
| `database.py` | Motor + Beanie 초기화, `KnowledgeBase`, `PromptTemplate`, `Document` 등록 |
| `milvus.py` | Milvus 연결, 컬렉션 생성/관리 |
| `fuseki.py` | Fuseki(SPARQL) 클라이언트 |
| `neo4j_client.py` | Neo4j 클라이언트 |
| `websocket_manager.py` | WebSocket 접속/푸시 관리 |

### 3.4 모델 (`app/models/`)

- **Beanie ODM** 사용 (MongoDB).
- `knowledge_base.py`: KB 메타데이터, chunking_config, graph_backend, pipeline_config 등
- `document.py`: 문서 메타데이터, status, pipeline_status, pipeline_metadata, 파일 경로 등
- `prompt.py`: 프롬프트 템플릿 저장

### 3.5 서비스 레이어 (`app/services/`)

- **embedding.py**: 임베딩 생성
- **ner.py**: NER 기반 필터/보조
- **ingestion/**  
  - **ingest_client.py**: Ingest Service HTTP 클라이언트 — `create_ingest_job()` 등
  - **service.py**: 레거시 문서 처리(청킹, 임베딩, 그래프 등). **현재 문서 업로드 플로우는 Ingest Service 위주**
  - **text_splitter.py**, **graph.py**, **graph_postprocessor.py**, **entity_store.py** 등: 청킹/그래프 후처리
- **retrieval/**  
  - **pipeline_executor.py**: 검색 파이프라인 순차 실행 (ann, bm25, brute_force, graph, rerank, ner_filter)
  - **vector.py**, **keyword.py**, **hybrid.py**, **two_stage.py**, **reranker.py**: 전략별 구현
  - **graph_backends/**  
    - **factory.py**: `GraphBackendFactory.get_backend("ontology"|"neo4j")` → Fuseki / Neo4j
    - **fuseki.py**, **neo4j.py**: 백엔드별 구현
  - **sparql_generator.py**, **cypher_generator.py**: 쿼리 생성

## 4. 문서 업로드 → 인제스션 데이터 흐름

1. **Frontend**  
   `docApi.upload(kbId, file, config)` → `POST /api/knowledge-bases/{kb_id}/documents` (FormData)
2. **Backend `document.py`**  
   - KB 조회, 동일 filename이면 기존 문서 재사용(덮어쓰기)
   - 파일을 `SHARED_STORAGE_PATH/kb_id/{doc_id}_{filename}`에 저장
   - 문서 레코드 생성/갱신 후 **BackgroundTasks**로 `ingest_client.create_ingest_job(...)` 호출
3. **Ingest Service** (`ingest_service/app/api/ingest.py`)  
   - `POST /api/ingest` 수신
   - `app.core.pipeline`의 인제스션 파이프라인 실행:  
     텍스트 추출 → (선택) 텍스트 정제 → 청킹 → 임베딩 → (선택) 그래프 추출/정규화/추론 → Milvus/Fuseki·Neo4j 적재
   - 완료/실패 시 callback 등으로 Backend에 알릴 수 있음
4. **상태 알림**  
   Backend의 WebSocket(`/api/ws/{kb_id}`)으로 프론트에 처리 상태 전달

정리하면, **실제 “문서 → 청크 → 벡터/그래프” 파이프라인은 Ingest Service**에 있고, Backend는 **오케스트레이션 + 메타저장 + WebSocket**을 담당합니다.

## 5. 검색(Retrieval) 데이터 흐름

1. **Frontend**  
   `retrievalApi.retrieve(kbId, { query, top_k, strategy, pipeline, ... })`  
   → `POST /api/knowledge-bases/{kb_id}/retrieve`
2. **Backend `retrieval.py`**  
   - `RetrievalRequest` 해석
   - **pipeline**이 있으면 `PipelineExecutor.execute()` 사용  
     - 단계: ann → bm25 → brute_force → graph → rerank → ner_filter 등
   - pipeline 없으면 기존 전략별 로직(strategy: ann/keyword/2-stage/hybrid 등) 사용
3. **Graph 사용 시**  
   - `GraphBackendFactory.get_backend(kb.graph_backend)`  
     - `"ontology"` 또는 `"fuseki"` → FusekiBackend  
     - `"neo4j"` → Neo4jBackend
   - 각 백엔드에서 SPARQL/Cypher 생성·실행 후 청크 ID/메타데이터 반환
4. **응답**  
   청크 목록 + score, metadata, execution_log 등

## 6. Ingest Service 구조

- **진입점**: `ingest_service/app/main.py` — FastAPI, `/api` 아래에 ingest 라우터 마운트
- **API**: `app/api/ingest.py`  
  - `POST /api/ingest`: IngestRequest 수신 → 파이프라인 실행
  - Job 목록/상태 등 부가 API 있을 수 있음
- **코어**  
  - **pipeline**: 텍스트 추출, 청킹, 임베딩, 그래프 추출, 정규화, 추론, Milvus/Fuseki/Neo4j 적재
  - **fuseki_connector.py**, **neo4j_connector.py**, **milvus_connector.py**: 각 스토어 연동
  - **text_cleaner.py**, **entity_normalizer.py** 등: 전처리/후처리

자세한 단계는 `docs/architecture/ingestion_pipeline_reference.md`를 참고하면 됩니다.

## 7. 프론트엔드 구조

- **라우팅** (`App.tsx`):  
  - `/` → Dashboard  
  - `/kb/:id` → KnowledgeBaseDetail  
  - `/graph-viewer` → GraphViewer
- **API 사용처**  
  - `src/services/api.ts`:  
    - `kbApi`: list, create, get, delete, promote, extraction-rules/prompt, query-prompt, getTriples, getChunk 등  
    - `docApi`: list, upload, delete, getChunks, updateChunk, update, updatePipelineStatus, getPipelineData  
    - `retrievalApi`: retrieve 등
- **페이지**  
  - **Dashboard**: KB 목록, 생성  
  - **KnowledgeBaseDetail**: 문서 탭, 설정, 테스트(검색) 탭 등  
  - **GraphViewer**: 그래프/트리플 시각화
- **컴포넌트**  
  - 문서/업로드/설정/검색 결과/파이프라인 빌더/모달 등이 `src/components/`에 위치

## 8. 설정·환경 요약

- **Backend**  
  - `backend/app/core/config.py`: MongoDB, Milvus, Fuseki, Neo4j, OpenAI, **INGEST_SERVICE_URL**, **SHARED_STORAGE_PATH**
  - 실행: `uvicorn main:app --reload --host 0.0.0.0 --port 8000` (디렉터리는 `backend/`)
- **Ingest Service**  
  - 기본 포트 8001, Docker에서는 `INGEST_SERVICE_URL`로 백엔드가 호출
- **공유 스토리지**  
  - `SHARED_STORAGE_PATH`(기본 `data/uploads`)를 Backend와 Ingest Service가 같이 사용해야 함 (동일 볼륨 또는 공유 경로).

## 9. 확장 시 참고 포인트

- **새 검색 단계 추가**: `app/services/retrieval/pipeline_executor.py`의 `_handlers`와 해당 `_execute_*` 메서드
- **그래프 백엔드 추가**: `graph_backends/base.py` 인터페이스 구현 후 `factory.py`에 등록
- **문서 처리 단계 변경**: Ingest Service의 `app/core/pipeline.py` 및 관련 core 모듈
- **새 API**: `app/api/`에 라우터 추가 후 `main.py`에서 `include_router`
- **프론트 새 페이지/기능**: `App.tsx` 라우트 추가, `api.ts`에 필요한 API 메서드 추가

이 문서는 분석 시점 기준 구조를 담고 있으며, 구현이 바뀌면 해당 모듈/경로를 기준으로 함께 수정하는 것을 권장합니다.
